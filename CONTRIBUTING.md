# المساهمة في SLCD

شكراً لاهتمامك بالمساهمة في **SLCD**! هذا الملف يشرح كيفية إعداد بيئة التطوير، معايير الكود، وطريقة تقديم التغييرات.

---

## 📋 قبل البدء

- تأكد من قراءة [README.md](./README.md) لفهم بنية المشروع (`:core` و `:slcd-app`) وصيغة `.dlof`.
- بالمساهمة في هذا المستودع، أنت توافق على ترخيص مساهمتك بموجب [رخصة Apache 2.0](./LICENSE) الخاصة بالمشروع (انظر البند 5 من الرخصة).
- تحقّق من [Issues](../../issues) الحالية قبل البدء بعمل كبير، لتفادي ازدواجية الجهد. للتغييرات الكبيرة (ميزة جديدة، إعادة هيكلة)، يُفضّل فتح Issue للنقاش أولاً.

---

## 🛠️ إعداد بيئة التطوير

### المتطلبات

| الأداة | الإصدار |
|---|---|
| JDK | 17 |
| Gradle | 8.7 |
| Android Studio | أحدث إصدار مستقر (Ladybug أو أحدث) |
| Compile / Target SDK | 34 |
| Min SDK | 26 (`:slcd-app`) / 24 (`:core`) |

### الاستنساخ والبناء

```bash
git clone https://github.com/<org>/slcd.git
cd slcd
gradle :slcd-app:assembleDebug --stacktrace
```

> لا يحتوي المستودع على `gradlew` مرفق حالياً؛ استخدم `gradle` مثبَّتاً محلياً، أو ولّد الـ wrapper بـ:
> ```bash
> gradle wrapper --gradle-version 8.7
> ```

### تشغيل الفحوصات محلياً قبل الدفع

```bash
gradle :slcd-app:lintDebug --stacktrace
gradle :slcd-app:testDebugUnitTest :core:testDebugUnitTest --stacktrace
```

يجب أن تمر هذه الأوامر دون أخطاء قبل فتح أي Pull Request — نفس الفحوصات تُشغَّل تلقائياً عبر [GitHub Actions](./.github/workflows/build_SLCD.yml).

## 🌳 بنية الفروع (Branching)

- الفرع `main` محمي: لا دفع مباشر (direct push)، فقط عبر Pull Requests.
- سمِّ فروعك بنمط واضح:
  - `feature/<وصف-مختصر>` لميزة جديدة
  - `fix/<وصف-مختصر>` لإصلاح خلل
  - `chore/<وصف-مختصر>` لأعمال صيانة (تبعيات، توثيق، CI)

## ✍️ معايير الكود

- **Kotlin** مع اتباع [Kotlin Coding Conventions](https://kotlinlang.org/docs/coding-conventions.html) الرسمية.
- **Jetpack Compose**: `@Composable` بأسماء PascalCase، وتفضيل الدوال الصغيرة القابلة لإعادة الاستخدام على شاشات ضخمة أحادية الكتلة.
- حافظ على الفصل القائم: أي منطق متعلق بتحليل/تحقق/تشفير صيغة `dlof` يذهب إلى `:core`، لا إلى `:slcd-app` — الوحدة الأخيرة يجب أن تبقى واجهة عرض فقط تعتمد على `:core`.
- التعليقات التوضيحية للأقسام المعقدة تُكتب بالعربية اتّساقاً مع بقية المشروع؛ أسماء الدوال والمتغيرات بالإنجليزية.
- لا تُدخل أي اعتمادية (dependency) خارجية جديدة دون تبرير واضح في وصف الـ PR (الحجم، الترخيص، الحاجة الفعلية).

---

## ✅ قبل فتح Pull Request

1. تأكد أن `lintDebug` و `testDebugUnitTest` يمران بنجاح محلياً.
2. اكتب رسالة commit واضحة تصف **ماذا** و**لماذا** (ليس فقط "تعديل").
3. لأي تغيير على سلوك القارئ أو الصيغة، أضف/حدّث اختبارات الوحدة المرتبطة.
4. صف في وصف الـ PR: المشكلة التي يحلّها، طريقة الاختبار، وأي لقطات شاشة إن كان تغييراً في الواجهة.
5. اربط الـ PR بأي Issue ذي صلة عبر `Closes #رقم`.

سيقوم [Build SLCD workflow](./.github/workflows/build_SLCD.yml) تلقائياً بتشغيل Lint والاختبارات وبناء APK تجريبي عند فتح الـ PR.

## 🐛 الإبلاغ عن خلل

عند فتح Issue لخلل، رجاءً ضمّن:

- خطوات إعادة الإنتاج (Reproduction steps)
- السلوك المتوقَّع مقابل الفعلي
- إصدار Android والجهاز
- سجل الخطأ (Logcat) إن وُجد
